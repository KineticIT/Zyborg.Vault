Write-VltData

NAME

Write-VltData

SYNOPSIS

Write data (secrets or configuration) into Vault to a backend provider logically mounted at a specified path.

SYNTAX

Write-Data [-Path] <string> [-Key] <string[]> [-Value] <object[]> [-KeepSecretWrapper <SwitchParameter>] [-VaultAddress <string>] [-VaultProfile <string>] [-VaultSession <object>] [-VaultToken <string>] [<CommonParameters>]

Write-Data [-Paths] <string[]> [-Data] <Hashtable> [-KeepSecretWrapper <SwitchParameter>] [-VaultAddress <string>] [-VaultProfile <string>] [-VaultSession <object>] [-VaultToken <string>] [<CommonParameters>]

Write-Data [-Paths] <string[]> [-NoValues] <SwitchParameter> [-KeepSecretWrapper <SwitchParameter>] [-VaultAddress <string>] [-VaultProfile <string>] [-VaultSession <object>] [-VaultToken <string>] [<CommonParameters>]

DESCRIPTION

There are 3 different Parameter Sets corresponding to 3 typical usages of this cmdlet.

In the first (default) usage, a single vault path is specified and one or more key-value pairs are provided. The Keys are all provided as an array, and the values are all provided as an array, and the length of these two arrays must be equal. Additionally, the Keys and Values can be provided from the pipeline using objects with the corresponding property names.

In the second and third usages, either a single map of key-values is provided or no values are provided at all using the -NoValues switch (some vault backends support writing no value to trigger an action or behavior). Multiple vault paths are specified into which all will receive the same set of data values. Tha Paths can be provided from the pipeline.

The behavior of the write is determined by the secret provider mounted at the given path. For example, writing to "aws/policy/ops" will create an "ops" IAM policy for the AWS secret provider (configuration), but writing to "consul/foo" will write ta value directly into Consul at that key.
